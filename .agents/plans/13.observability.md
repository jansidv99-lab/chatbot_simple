# Module 13 — Observability (Prometheus + Grafana)

⚠️ **Medium**

## Context

Module 7 delivered LLM-level tracing via Arize Phoenix (OpenTelemetry spans on every graph node and Streamlit page). Module 13 adds the **infrastructure metrics** layer: Prometheus scrapes FastAPI HTTP metrics + custom business counters; Grafana visualises them in dashboards.

```
Application (FastAPI)  →  /metrics endpoint  →  Prometheus  →  Grafana dashboards
LLM Traces             →  Phoenix (already complete, Module 7)
```

## What Already Exists

- Phoenix deployed in Helm (`phoenix-deployment.yaml`, `phoenix-service.yaml`)
- All 9 LangGraph nodes + 3 Streamlit pages instrumented with OTel spans
- `arize-phoenix-otel`, `opentelemetry-instrumentation-ollama` already in `requirements.txt`

## What Gets Added

### Application layer (`api/`)

| Metric name | Type | Labels | Meaning |
|---|---|---|---|
| `http_requests_total` | Counter | method, handler, status | All FastAPI HTTP requests (auto via instrumentator) |
| `http_request_duration_seconds` | Histogram | method, handler | Latency per endpoint (auto via instrumentator) |
| `chatbot_chat_requests_total` | Counter | — | Chat requests that passed rate limiting |
| `chatbot_analytics_requests_total` | Counter | — | Analytics requests that passed rate limiting |
| `chatbot_rate_limit_hits_total` | Counter | endpoint | 429 responses emitted by rate limiter |

### Infrastructure layer (Helm)

Prometheus + Grafana deployed as simple ClusterIP pods (same pattern as Redis, PostgreSQL, Phoenix).

## Architecture

```
FastAPI :8000/metrics
        ↓
Prometheus (scrape every 15 s)
        ↓
Grafana  :3000  ←── pre-provisioned datasource (ConfigMap)
```

### 2. NEW `api/metrics.py` — 3 custom counters
```python
from prometheus_client import Counter

CHAT_REQUESTS = Counter("chatbot_chat_requests_total", "Chat requests processed")
ANALYTICS_REQUESTS = Counter("chatbot_analytics_requests_total", "Analytics requests processed")
RATE_LIMIT_HITS = Counter("chatbot_rate_limit_hits_total", "Rate limit 429s", ["endpoint"])
```

### 3. `api/main.py` — add instrumentator (2 lines)
After the existing `app.include_router(...)` calls, add:
```python
from prometheus_fastapi_instrumentator import Instrumentator
Instrumentator(excluded_handlers=["/metrics", "/health"]).instrument(app).expose(app)
```
`excluded_handlers` prevents `/metrics` and `/health` themselves from polluting the request counters.

### 4. `api/rate_limit.py` — increment hit counter before raising 429
```python
from api.metrics import RATE_LIMIT_HITS
# In SlidingWindowRateLimiter.check(), before `raise HTTPException(429, ...)`:
RATE_LIMIT_HITS.labels(endpoint=key.split(":")[0]).inc()
```
`key.split(":")[0]` extracts the endpoint prefix (`chat`, `analytics`, `login`) from keys like `chat:42`.

### 5. `api/routers/chat.py` — increment chat counter (2 lines)
```python
from api.metrics import CHAT_REQUESTS
```

## Known Limitations

- FastAPI `/metrics` scraping in k8s requires `api/` to be added to the Dockerfile (existing Module 11 gap). Until then, Prometheus self-scrape works but FastAPI metrics are local-dev only.
- Grafana and Prometheus use ephemeral storage (no PVC) — dashboards saved in the UI are lost on pod restart. Use ConfigMap provisioning for any dashboard worth keeping.
- `adminPassword` is plaintext in `values.yaml` — same known debt as `PG_PASSWORD` and `jwtSecretKey`.
