# Module 9 — Multi-Agent F&O Data Analysis (LangGraph)

🔴 **Complex**

## Architecture

### Agent Flow

```mermaid
flowchart TD
    START([START]) --> supervisor

    supervisor[Supervisor\nIs question answerable\nfrom F&O tables?]
    supervisor -->|answerable| schema_agent
    supervisor -->|off-topic| END_OT([END\nrejection message])

    schema_agent[Schema Agent\nBuild schema context\nfrom known DDL] --> sql_planner

    sql_planner[SQL Planner\nLLM generates SQL\nfrom question + schema] --> sql_validator

    sql_validator[SQL Validator\nRun EXPLAIN via psycopg2\nto check syntax]
    sql_validator -->|valid| execute_sql
    sql_validator -->|invalid| clarification

    execute_sql[Execute SQL\nRun query via get_connection\nCollect rows]
    execute_sql -->|data found| analytics
    execute_sql -->|no data| clarification

    clarification[Clarification Agent\nLLM rewrites SQL\nusing error context\nretry_count ++]
    clarification -->|retry_count < 3| sql_validator
    clarification -->|retry_count >= 3| response_formatter

    analytics[Analytics Agent\nLLM interprets results\nproduces narrative] --> validation

    validation[Validation Node\nEnsure analysis is\nnon-empty + references numbers]
    validation -->|pass| response_formatter
    validation -->|fail and retry < 3| clarification

    response_formatter[Response Formatter\nAssemble final markdown\nresponse] --> END([END])
```

### Node Summary

| Node | LLM? | DB? | Key output |
|---|---|---|---|
| supervisor | ✅ | ❌ | route decision |
| schema_agent | ❌ | ✅ `list_tables()` | `schema_context` |
| sql_planner | ✅ | ❌ | `sql_query` |
| sql_validator | ❌ | ✅ `EXPLAIN` | `sql_valid`, `validation_error` |
| execute_sql | ❌ | ✅ SELECT | `query_results`, `data_found` |
| clarification | ✅ | ❌ | revised `sql_query` |
| analytics | ✅ | ❌ | `analysis` narrative |
| validation | ❌ | ❌ | pass / fail route |
| response_formatter | ❌ | ❌ | `final_response` |

### State Schema (`AnalysisState` TypedDict)

```
question, schema_context, sql_query, sql_valid, validation_error,
query_results, data_found, analysis, final_response, retry_count
```

### LLM

`ChatOllama` (`langchain-ollama`) — reads `OLLAMA_HOST` and `MODEL_NAME` env vars.
Same Ollama server already running; no new infrastructure.

### DB

All DB calls reuse `get_connection()` and `list_tables()` from `ingestion/db.py`.

### Tracing

`sql_planner`, `clarification_agent`, `analytics_agent` wrapped in OpenTelemetry spans
(`openinference.span.kind = "LLM"`) — same pattern as `app.py`.

## Known Limitations

- SQL Planner may hallucinate column names on complex JOINs; Clarification Agent
  catches these via the EXPLAIN validator
- Analytics narrative quality depends on the Ollama model in use
- No streaming — `graph.invoke()` runs synchronously; spinner shown during execution
