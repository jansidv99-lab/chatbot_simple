## Architecture

```
Eval Results page (pages/evals.py)
  ├── model_input = "gemma4:e2b"  →  run_evals(dataset, model_name="gemma4:e2b")  →  save
  └── model_input = "qwen3:1.7b"  →  run_evals(dataset, model_name="qwen3:1.7b")  →  save

A/B Comparison page (pages/ab_test.py)
  ├── select run A (model A, run_id_a)
  ├── select run B (model B, run_id_b)
  └── compare_runs(run_id_a, model_a, run_id_b, model_b)
        ├── get_run_results(run_id_a)  →  dict keyed by question_id
        ├── get_run_results(run_id_b)  →  dict keyed by question_id
        ├── align on common question_ids
        └── QuestionComparison.winner = sum_a > sum_b ? "A" : sum_b > sum_a ? "B" : "tie"
```

## Model Override Mechanism

`run_evals(dataset, model_name="qwen3:1.7b")` does:
1. `os.environ["MODEL_NAME"] = "qwen3:1.7b"`
2. `_get_llm.cache_clear()` — forces next `graph.invoke()` to create a new ChatOllama instance
3. Runs all questions, records `model_name` in EvalRun
4. No env var restore — next call with a different model_name will override again

**Single-user caveat**: env var change is process-wide. Acceptable for local Streamlit.

## Judge Decoupling

`judge.py` no longer imports from `agents.graph`. It uses `_get_judge_llm(model)` with its own `lru_cache(maxsize=4)`. The judge model is resolved as:
```python
judge_model = os.environ.get("JUDGE_MODEL") or os.environ.get("MODEL_NAME", "gemma4:e2b")
```
To pin a stable judge across model comparisons, set `JUDGE_MODEL=gemma4:e2b` in `.env`.

## Running an A/B Comparison

```powershell
# CLI — run two models sequentially
python -m evals.runner qwen3:1.7b
python -m evals.runner gemma4:e2b

# Streamlit
streamlit run app.py
# → Eval Results: change model name input, click Run Evals twice
# → A/B Model Comparison: select the two runs, view side-by-side
```

## Known Constraints

- Model must be pre-pulled in Ollama before running. Failures captured in `EvalResult.error`.
- `JUDGE_MODEL` unset → judge uses same model as tested run (directionally consistent but not calibrated).
- No schema changes — uses existing `eval_runs` / `eval_results` tables from Module 16.
