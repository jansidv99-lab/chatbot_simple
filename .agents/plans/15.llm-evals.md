## Architecture

```
Golden Dataset (evals/golden_dataset.json)
  ↓
runner.run_evals(dataset)
  ├── per question: graph.invoke(initial_state)      ← real DB + Ollama
  ├── per question: judge.score(q, response, rows)   ← Ollama LLM-as-judge
  └── EvalRun { EvalResult × 20 }
        ↓
evals/db.py → eval_runs + eval_results tables (PostgreSQL)
        ↓
pages/evals.py → Streamlit UI (run history + drill-down)
```

## Golden Dataset Categories

| Category | Count | Examples |
|---|---|---|
| aggregation | 8 | total P&L, total brokerage, avg trade price |
| ranking | 6 | highest profit symbol, top 5 by quantity |
| filtering | 5 | negative P&L symbols, recent week trades |
| multi-table | 3 | trades + P&L join, buy/sell ratio |

All questions are **relative** (no hardcoded expected values) — work with any uploaded F&O data.

## Judge Scoring

Uses `_get_llm()` from `agents/graph.py` (same ChatOllama singleton). Scores 3 dimensions 1–5:
- **relevance** — does the response answer the question?
- **completeness** — includes specific numbers and context?
- **groundedness** — claims supported by SQL results, no hallucination?

`passed = all scores >= 3`. Fails open on JSON parse errors (defaults to 3).

## Database Schema

```sql
eval_runs(run_id, run_at, model_name, total_questions, pass_rate, data_found_rate,
          sql_valid_rate, avg_latency_s, avg_relevance, avg_completeness, avg_groundedness)

eval_results(run_id→eval_runs, question_id, question, final_response, sql_query,
             data_found, sql_valid, retry_count_sql, retry_count_analysis,
             latency_s, judge_relevance, judge_completeness, judge_groundedness,
             judge_passed, error, evaluated_at)
```

## No New Dependencies

Reuses: `_get_llm()` + `HumanMessage` (agents/graph.py), `get_connection()` (ingestion/db.py), `langchain_core`, `psycopg2`. No ragas/deepeval/sentence-transformers needed.

## Running Evals

```powershell

# Streamlit UI
streamlit run app.py
# → navigate to "Eval Results" in sidebar → click "Run Evals"
```

## Known Constraints

- **Duration**: 20 questions × (pipeline + judge) ≈ 20–40 min. Streamlit blocks during run; use CLI for long runs.
- **Data prerequisite**: Upload data via `pages/upload.py` first. Empty DB → `data_found=False` on all questions (useful signal, not a crash).
- **Judge calibration**: gemma4:e2b scores are directionally useful for relative comparisons across runs, not absolute quality claims.
- **No CI linting**: `evals/` not in ruff scope (CI lints `app.py pages/ ingestion/` only).
