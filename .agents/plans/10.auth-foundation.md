## Architecture

```
User
 ↓
pages/login.py  ──► auth/users.py ──► PostgreSQL (users table)
     │                    │
     │           auth/passwords.py (bcrypt hash_password / verify_password)
     ↓
auth/session.py ──► auth/tokens.py (PyJWT create / decode)
     │
     ↓  stores in st.session_state
{auth_access_token, auth_refresh_token, auth_user}
     ↓
require_auth()  ←── called at top of app.py, pages/upload.py, pages/analytics.py
```

## Users Table

```sql
CREATE TABLE IF NOT EXISTS users (
    id             SERIAL        PRIMARY KEY,
    username       VARCHAR(50)   NOT NULL UNIQUE,
    email          VARCHAR(255)  NOT NULL UNIQUE,
    password_hash  VARCHAR(72)   NOT NULL,
    created_at     TIMESTAMPTZ   NOT NULL DEFAULT NOW(),
    is_active      BOOLEAN       NOT NULL DEFAULT TRUE
);
```

## Token Design

| Token | Expiry | Payload | Purpose |
|---|---|---|---|
| Access | 30 min | `sub`, `username`, `type="access"`, `iat`, `exp` | Page access auth |
| Refresh | 7 days | `sub`, `type="refresh"`, `iat`, `exp` | Auto-renew access token |

Both stored in `st.session_state`. On access token expiry, `_try_refresh()` mints a new access token using the refresh token + a DB check (`is_active`).

## Page Gating Pattern

```python
# identical in app.py, pages/upload.py, pages/analytics.py
init_session_state()
require_auth()   # st.error + st.stop() if not authenticated
```

## Known Limitations

- `JWT_SECRET_KEY` is plaintext in `values.yaml` — move to a Kubernetes Secret before any shared/production environment (same known debt as `PG_PASSWORD`)
- Streamlit sessions are ephemeral: tokens are lost on browser refresh, requiring re-login
- No account lockout after N failed attempts (can be added in a future hardening module)
- `auth/` is excluded from CI lint scope (matches the existing `agents/` exclusion)
