## Architecture

```
Incoming request
  ↓
rate_limit.check(...)
  ↓
cache_get(prefix, [query])
  ├── _embed(query)  →  Ollama nomic-embed-text  →  768-dim vector
  ├── SCAN Redis  sem:{prefix}:*
  ├── for each entry: cosine_similarity(query_vec, stored_vec)
  ├── best_score >= 0.90  →  return cached response  (LLM skipped)
  └── best_score < 0.90   →  cache miss → LLM call → cache_set → return
```

### What gets cached

| Endpoint | Cache condition | TTL | Cache key scope |
|---|---|---|---|
| `POST /analytics/` | Always (question only, no history) | 3600 s | Shared across all users |
| `POST /chat/` | Only when `history == []` | 300 s | Shared across all users |

Conversational chat turns (`history != []`) are never cached — same message in a different context should produce a different response.

## Redis Data Structure

Each cache entry is stored at `sem:{prefix}:{uuid4}` → JSON string:

```json
{"embedding": [0.12, -0.34, ...], "response": "<cached response text>"}
```

- TTL is set per-entry with `SETEX` (atomic SET + EXPIRE)
- No index set — `SCAN sem:{prefix}:*` discovers live entries; expired keys are skipped automatically
- Each prefix (`analytics`, `chat`) forms its own keyspace

### Inspecting cache entries (redis-cli)

```powershell
# In-cluster
kubectl exec deployment/chatbot-redis -- redis-cli

# List all semantic cache keys
KEYS sem:*

# Count analytics cache entries
SCAN 0 MATCH sem:analytics:* COUNT 100

# Inspect one entry (large — embedding is 768 floats)
GET sem:analytics:<uuid>

# Check TTL remaining
TTL sem:analytics:<uuid>

# Clear the entire semantic cache (dev only)
KEYS sem:* | ForEach-Object { kubectl exec deployment/chatbot-redis -- redis-cli DEL $_ }
# Or from inside redis-cli:
# EVAL "return redis.call('del', unpack(redis.call('keys', ARGV[1])))" 0 sem:*
```

## Similarity Algorithm

Cosine similarity via numpy:

```
score = dot(va, vb) / (||va|| * ||vb||)
```

- Threshold: `CACHE_SIM_THRESHOLD` env var (default `0.90`)
- `0.90` catches paraphrases ("show my P&L" ≈ "what is my profit and loss")
- `1.0` = exact same text after normalization
- Lower threshold → more hits, higher risk of returning a wrong cached answer

## Embedding Model

`nomic-embed-text` (default) — 768-dimensional, fast, ~274 MB download.

Input is normalized before embedding: `text.strip().lower()`.

## Fail-Open Behavior

Both `cache_get` and `cache_set` fail open:
- Embedding error → `cache_get` returns `None` (proceeds to LLM), `cache_set` silently skips
- Redis error → same: log `WARNING`, allow request through

`CHAT_REQUESTS.inc()` fires unconditionally (cache hit and miss). `ANALYTICS_REQUESTS.inc()` fires only on cache miss (actual LLM call).

## New Env Vars

| Variable | Default | Purpose |
|---|---|---|
| `EMBED_MODEL` | `nomic-embed-text` | Ollama model used for embedding queries |
| `CACHE_SIM_THRESHOLD` | `0.90` | Minimum cosine similarity to consider a cache hit |

## Known Limitations

- `SCAN` is O(N) over all Redis keys. For large keyspaces (thousands of cache entries), consider switching to RedisSearch vector index (`redis/redis-stack` image) for O(log N) ANN lookup.
- Embedding adds ~50–200ms latency per request (Ollama inference). On cache hit this replaces 5–30s of LangGraph pipeline time — trade-off is clearly positive for analytics.
- Cache is shared across users. If different users have different data access rights in the future, per-user cache keys would be needed.
- `nomic-embed-text` must be pulled on the Ollama host before the API starts. If missing, all requests fail open (embedding error → no caching, LLM still runs normally).
- Chat SSE replay serves the full cached payload as a single SSE event instead of token-by-token. The Streamlit client is unaffected (it concatenates all tokens), but the streaming UX is lost on a cache hit.
