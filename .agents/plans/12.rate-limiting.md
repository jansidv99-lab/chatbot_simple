## Rate Limits

| Endpoint | Key | Limit | Algorithm |
|---|---|---|---|
| `POST /chat/` | `chat:{user_id}` | 20 req / 60 s | Sliding Window |
| `POST /analytics/` | `analytics:{user_id}` | 5 req / 60 s | Sliding Window |
| `POST /auth/login` | `login:{client_ip}` | 10 req / 60 s | Sliding Window |

`user_id` = `_user["sub"]` from decoded JWT (stable numeric string). Login uses `request.client.host` (IP-based, no JWT yet at that point).

## Sliding Window Algorithm (ZSET-based)

```
1. ZREMRANGEBYSCORE key "-inf" (now - window)   remove expired entries
2. ZCARD key                                      count remaining (pipelined with step 1)
3. if count >= limit Ôćĺ HTTP 429
4. ZADD key {uuid4: now}                         add this request
5. EXPIRE key window                             reset TTL
```

Members use `uuid.uuid4()` (not timestamp strings) to guarantee uniqueness even when multiple calls occur within the same clock tick ÔÇö timestamp-based members collide on Windows where `time.time()` resolution is ~100 ns.

**Atomicity note:** Steps 1ÔÇô2 are pipelined but step 4 is a separate command. A TOCTOU race can allow N+1 requests under high concurrency. Production upgrade: replace with a Lua script via `redis.register_script()` to make check + add atomic.

## Token Bucket Algorithm (HASH-based, educational)

```
Redis HASH stores: {tokens: float, last_refill: float}

1. hgetall key
2. tokens = min(capacity, stored_tokens + elapsed * refill_rate)
3. if tokens < 1 Ôćĺ HTTP 429
4. hset key {tokens: tokens-1, last_refill: now}
5. expire key (capacity / refill_rate) + 1
```

Not wired to any route ÔÇö demonstrates the algorithm. Same atomicity caveat as sliding window.

## Fail-Open Behavior

Both classes catch `redis.RedisError` (covers `ConnectionError`, `TimeoutError`, `ResponseError`), log a `WARNING`, and return normally ÔÇö allowing the request through. The `except HTTPException: raise` guard appears first to prevent 429s from being swallowed.

## Known Limitations

- `REDIS_HOST`/`REDIS_PORT` are read at import time (module-level `redis.Redis()` call). Changing env vars after process start has no effect.
- Login rate limit keys are IP-based. Behind a reverse proxy or Kubernetes ingress, all requests may share the same `request.client.host` (the proxy IP) unless `X-Forwarded-For` is extracted.
- Redis data is ephemeral ÔÇö pod restarts reset all rate limit counters (by design; counters are not business-critical state).
- `api/` still not in the Dockerfile ÔÇö rate limiting is only active in local-dev mode (see Module 11 known limitation).
